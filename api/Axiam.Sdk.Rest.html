AuthzRestClient

REST authorization client (CONTRACT.md §1, FND-04): CheckAccessAsync/ CanAsync/BatchCheckAsync over POST /api/v1/authz/check and POST /api/v1/authz/check/batch (mirrors the gRPC CheckAccess/BatchCheckAccess semantics a later plan wires up over the same shared session). Exposed as AxiamClient.Authz.

AuthzRestClient.AccessCheck

A single authorization check request item for BatchCheckAsync(IEnumerable<AccessCheck>, CancellationToken).

AxiamHttpClientFactory

Builds the SDK-owned HttpClient (D-09 default path) and configures the IHttpClientFactory-compatible alternative path (D-09 alt path, wired up by a later plan's AddAxiam()). Both paths guarantee the CONTRACT.md §4 persistent cookie jar and the §6 no-TLS-bypass policy — a caller-supplied handler can never silently drop either guarantee (client-override safety, Java D-27 analog).

AxiamHttpMessageHandler

Cross-cutting REST transport concerns shared by every request an AxiamClient sends (CONTRACT.md §3/§5/§9): injects X-Tenant-Id (§5) and, when a session exists, Authorization: Bearer <access> read from the shared cookie jar; reads the axiam_csrf cookie the server sets and echoes it as the X-CSRF-Token request header on subsequent state-changing requests (§3 cookie double-submit — the real AXIAM server only ever delivers the CSRF value via a cookie, never a response header, so the cookie-jar read is the load-bearing path; the response-header capture is kept as defense-in-depth); and drives a single reactive 401→refresh→retry through the shared RefreshGuard — never a second, independent guard, and never a retry loop (§9.3).
