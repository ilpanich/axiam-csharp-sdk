Builds an AuthorizationPolicy containing a single AxiamRequirement for either the structured policy name produced by AxiamAccessAttribute or the legacy policy name shaped "resource:action" (e.g. "documents:read") — the novel .NET-idiom file this phase introduces (D-08; no direct Spring/method-security equivalent, built from RESEARCH.md Pattern 5). Falls back to DefaultAuthorizationPolicyProvider for every other policy name, so consumer-defined policies (registered via the normal AddAuthorization(options => options.AddPolicy(...)) path) keep working completely unmodified.

GetPolicyAsync(string)

Recognizes two policy-name shapes and builds an AxiamRequirement-backed policy for either on the fly — the consumer never has to explicitly register these policies via AddAuthorization(options => options.AddPolicy(...)):

  • The structured form produced by AxiamAccessAttribute ("axiam::<action>::<resource>::<scope>::<param>"), carrying Scope and ResourceRouteParam through to the built requirement. Checked FIRST.
  • The legacy "resource:action"-shaped policy name (exactly one non-empty resource half and one non-empty action half, separated by a single ':', e.g. "documents:read") — kept fully supported for existing [Authorize(Policy = "resource:action")] consumers.

Any other policy name (including one containing no ':', or an explicitly pre-registered policy that happens to be named with a colon) falls through to DefaultAuthorizationPolicyProvider unmodified.
