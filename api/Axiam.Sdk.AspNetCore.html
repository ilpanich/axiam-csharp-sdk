Namespace Axiam.Sdk.AspNetCore

Classes

AxiamAccessAttribute

Declarative, per-endpoint AXIAM authorization attribute (CONTRACT.md §11 require_access(action, resource[, scope]) — C# maps this to [AxiamAccess(...)]). Sugar over the existing "resource:action" policy string mechanism (AxiamPolicyProvider/AxiamPolicyHandler): this attribute serializes its Action/Resource/ Scope/ResourceRouteParam into a structured policy name ("axiam::<action>::<resource>::<scope>::<param>") that AxiamPolicyProvider recognizes and parses ALONGSIDE the legacy single-colon "resource:action" form (which remains fully supported — see its own remarks). Because this type derives from AuthorizeAttribute, it can be placed directly on a controller/action ([AxiamAccess("read", "documents")]) or passed to RequireAuthorization(IAuthorizeData) for minimal APIs — ASP.NET Core's own authorization pipeline combines it into a policy via Policy exactly like any other AuthorizeAttribute.

AxiamAuthMiddleware

ASP.NET Core authentication middleware (D-06, CONTRACT.md §10 "ASP.NET Core" row: app.UseMiddleware<AxiamAuthMiddleware>() in Program.cs). Mirrors sdks/java/.../spring/AxiamAuthenticationFilter.java's authoritative sequence (extract → verify → exp-check → cross-tenant-check → inject-identity → 401 JSON on failure), adapted to ASP.NET Core's ClaimsPrincipal/User integration point — the direct analog of Java D-14's Spring SecurityContext integration.

AxiamAuthorizationMiddlewareResultHandler

Custom IAuthorizationMiddlewareResultHandler that writes the standardized JSON error body (CONTRACT.md §10/§11, PATTERNS.md "Standardized JSON error body") for every authorization outcome: an unauthenticated result (401), a "lacks permission" result (403 — AxiamPolicyHandler left an AxiamRequirement unsatisfied, i.e. AuthzError), an unresolvable resource id (400 invalid_request, CONTRACT.md §11.2.3), and a transport failure while checking access (503 authz_unavailable, fail-closed per CONTRACT.md §11.2.5). On success, the pipeline continues exactly like the framework default.

AxiamOptions

Typed Options-pattern configuration for AddAxiam(IServiceCollection, Action<AxiamOptions>)/ AddAxiamAspNetCore(IServiceCollection, Action<AxiamOptions>) (D-07). Configured via services.AddAxiamAspNetCore(options => { ... }) — properties are mutable (set, not init) because the .NET Options pattern configures an already-constructed instance through an Action<T> delegate.

AxiamPolicyHandler

Routes [Authorize(Policy="resource:action")] (e.g. "documents:read") into AxiamClient.Authz.CheckAccessAsync (D-08). No direct Spring analog — Spring's sibling SDK uses method-security annotations rather than an IAuthorizationHandler/policy-provider pair, so this is the most novel file in the phase (built from RESEARCH.md Pattern 5, reusing the AuthzError→403 JSON-body convention from AxiamAuthenticationFilter.java lines 102-111).

AxiamPolicyProvider

Builds an AuthorizationPolicy containing a single AxiamRequirement for either the structured policy name produced by AxiamAccessAttribute or the legacy policy name shaped "resource:action" (e.g. "documents:read") — the novel .NET-idiom file this phase introduces (D-08; no direct Spring/method-security equivalent, built from RESEARCH.md Pattern 5). Falls back to DefaultAuthorizationPolicyProvider for every other policy name, so consumer-defined policies (registered via the normal AddAuthorization(options => options.AddPolicy(...)) path) keep working completely unmodified.

AxiamRequirement

Authorization requirement carrying everything AxiamPolicyHandler needs to run a single AXIAM access check (CONTRACT.md §11): the resource/action halves (legacy "resource:action" policy-string form, e.g. "documents:read", or the structured AxiamAccessAttribute form — both funnel into this same type), plus the optional Scope and the ResourceRouteParam route value name used to resolve the target resource's UUID at request time. Built by AxiamPolicyProvider.

ServiceCollectionExtensions

IServiceCollection DI extensions (D-07): AddAxiam(IServiceCollection, Action<AxiamOptions>) registers the typed AxiamOptions plus a single shared AxiamClient; AddAxiamAspNetCore(IServiceCollection, Action<AxiamOptions>) additionally wires the policy-based authorization surface (D-08). The raw app.UseMiddleware<AxiamAuthMiddleware>() form (CONTRACT.md §10) remains available and works with either registration path.
