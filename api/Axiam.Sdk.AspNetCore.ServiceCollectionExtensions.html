Remarks

Every registration uses TryAdd* (never AddSingleton) so an explicit consumer registration made BEFORE calling these methods always wins — the .NET equivalent of Java's AxiamAutoConfiguration @ConditionalOnMissingBean precedence (PATTERNS.md "ServiceCollectionExtensions.cs"). This is exercised directly by the SC#3 integration test, which registers a test-seam AxiamClient (pointed at a fake transport) BEFORE calling AddAxiamAspNetCore(IServiceCollection, Action<AxiamOptions>).

Methods

AddAxiam(IServiceCollection, Action<AxiamOptions>)

Registers AxiamOptions (Options pattern) and a single shared AxiamClient singleton built from it. Does NOT register any authorization/middleware services — use AddAxiamAspNetCore(IServiceCollection, Action<AxiamOptions>) for the full ASP.NET Core integration.

AddAxiamAspNetCore(IServiceCollection, Action<AxiamOptions>)

Calls AddAxiam(IServiceCollection, Action<AxiamOptions>) and additionally registers the policy-based authorization surface (D-08): AxiamPolicyHandler, AxiamPolicyProvider, and AxiamAuthorizationMiddlewareResultHandler (standardized 401/403 JSON bodies). Also calls the framework's own AddAuthorization() so the base authorization services exist even if the consuming app never calls it itself — safe to call even when the app already has, since ASP.NET Core's own AddAuthorization() uses TryAdd* internally.
