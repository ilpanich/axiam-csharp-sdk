ConfigureFactoryHandler(SocketsHttpHandler, byte[]?, byte[]?)

Re-applies the SDK's cookie jar (§4) and pooled-connection lifetime settings onto a SocketsHttpHandler registered via IHttpClientFactory/ AddHttpClient (D-09 alt path, wired up by a later plan) — even if the caller supplied their own handler instance, client-override safety means the cookie jar can never be silently dropped (breaking post-login). The only SslOptions this method ever writes is the §6.1 mTLS client identity (ClientCertificates, what the client OFFERS) when a client certificate is configured — it never sets RemoteCertificateValidationCallback or otherwise weakens server verification.

CreatePrimaryHandler(byte[]?, byte[]?, byte[]?)

Builds the primary HttpClientHandler with UseCookies = true and a private CookieContainer (§4). When customCaPem is supplied, an ADDITIVE chain-trust-store callback is installed — it never returns true unconditionally (§6/ SC#4: no TLS-bypass surface exists anywhere in this SDK). Exposed separately from CreateOwned(byte[]?) so AxiamClient can compose this handler as the innermost link of its own AxiamHttpMessageHandler chain while still sharing the exact same cookie-jar/TLS configuration.

public static HttpClientHandler CreatePrimaryHandler(byte[]? customCaPem, byte[]? clientCertPem = null, byte[]? clientKeyPem = null)

Remarks

When clientCertPem/clientKeyPem are supplied (CONTRACT.md §6.1), the PEM client-certificate identity is added to ClientCertificates for mutual-TLS client authentication. This is a distinct code path from the additive custom-CA server-trust callback above: presenting a client certificate NEVER relaxes strict server verification (§6.1 rule 2), and this method installs no permissive server-validation delegate.
