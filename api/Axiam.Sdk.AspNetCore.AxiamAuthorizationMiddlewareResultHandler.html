Custom IAuthorizationMiddlewareResultHandler that writes the standardized JSON error body (CONTRACT.md §10/§11, PATTERNS.md "Standardized JSON error body") for every authorization outcome: an unauthenticated result (401), a "lacks permission" result (403 — AxiamPolicyHandler left an AxiamRequirement unsatisfied, i.e. AuthzError), an unresolvable resource id (400 invalid_request, CONTRACT.md §11.2.3), and a transport failure while checking access (503 authz_unavailable, fail-closed per CONTRACT.md §11.2.5). On success, the pipeline continues exactly like the framework default.

Remarks

This handler deliberately does NOT branch on Forbidden/Challenged. Neither this package nor a typical consumer registers an ASP.NET Core authentication scheme (AddAuthentication()/UseAuthentication()) — identity comes entirely from AxiamAuthMiddleware setting User directly. Because of that, Microsoft.AspNetCore.Authorization.Policy.PolicyEvaluator's internal authentication step always resolves to AuthenticateResult.NoResult() (Succeeded == false) since the evaluated policy carries no AuthenticationSchemes — which means EVERY non-success outcome, including an authenticated-but-forbidden one, would surface as Challenged, never Forbidden. Branching on User's own Identity.IsAuthenticated instead sidesteps that ambiguity entirely and is the authoritative signal: AxiamAuthMiddleware is the only code path that ever sets an authenticated identity, and it only does so after JwksVerifier signature+tenant verification succeeds.

HandleAsync(RequestDelegate, HttpContext, AuthorizationPolicy, PolicyAuthorizationResult)

On success, continues the pipeline exactly like the framework default. On failure, writes a standardized JSON error body, checking Axiam.Sdk.AspNetCore.AxiamPolicyHandler.OutcomeItemKey in context's Items FIRST — see that key's remarks for why this, rather than authorizeResult's own AuthorizationFailure, is the channel AxiamPolicyHandler actually reaches this handler through: 503 (authz_unavailable) for a transport failure while checking access, 400 (invalid_request) for an unresolvable resource id. Absent either signal, falls back to the §10 behavior: 403 (authorization_denied) when User is already authenticated (an AxiamRequirement was left unsatisfied by AxiamPolicyHandler), otherwise 401 (authentication_failed). See the type-level remarks for why this branches on User.Identity.IsAuthenticated rather than authorizeResult's own Forbidden/Challenged distinction.
