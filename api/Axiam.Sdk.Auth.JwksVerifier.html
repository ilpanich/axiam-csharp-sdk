Local EdDSA (Ed25519) JWKS verification (D-02, CONTRACT.md §10 local-verification fast path). .NET has no native Ed25519/EdDSA support anywhere in System.Security.Cryptography (confirmed research finding — dotnet/runtime #14741/#63174 remain unimplemented through .NET 10 GA), so this class uses BouncyCastle.Cryptography's Org.BouncyCastle.Crypto.Signers.Ed25519Signer / Org.BouncyCastle.Crypto.Parameters.Ed25519PublicKeyParameters directly — the single, well-vetted, verify-only crypto dependency D-02 permits.

Remarks

Security-critical invariants (T-21-06, T-21-07):

  • alg is pinned to "EdDSA" and checked BEFORE any key (kid) lookup — the token's own header is never trusted to select its verifier (alg-confusion defense).
  • AFTER signature verification succeeds, the tenant_id claim is checked against the caller-supplied expected tenant. The JWKS document is organization-wide, not tenant-scoped, so a valid signature alone never implies tenant authorization (Pitfall 3 — independently confirmed by every sibling SDK).
  • VerifyAsync(string, string, CancellationToken) NEVER throws for attacker-controlled input — every failure mode (bad alg, unknown kid, tampered/invalid signature, wrong tenant, expired token, malformed/non-base64/truncated token) returns null. This matches the AMQP HMAC verifier's fail-closed convention (Amqp/Hmac.cs).

VerifyAsync(string, string, CancellationToken)

Verifies jwt's EdDSA signature against the cached (or freshly fetched) org-wide JWKS AND checks the mandatory tenant_id claim against expectedTenantId. Returns the decoded claims payload on success; returns null for ANY failure. Never throws on malformed or attacker-controlled input — see the type-level remarks for the fail-closed contract.
